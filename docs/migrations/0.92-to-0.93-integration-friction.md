# 0.92 → 0.93 integration friction log (dogfood: web-app)

A record of integrating the **experimental `@quillmark/wasm` built from `main`**
into `tonguetoquill/web-app`, consuming `docs/migrations/0.92-to-0.93.md` as a
third-party would. The headline: the **JS/TS API surface is genuinely unchanged**
— integration needed no consumer code edits, typecheck stayed clean, and 178
wasm-touching tests passed against the new build. All real friction was in
**toolchain, versioning, and packaging**, not in the migrated API.

## What went smoothly (the guide held)

- The guide's "publicly unchanged" promises were accurate. `engine.render`,
  `engine.supportsCanvas`, `RenderSession`, `Quill`/`Document` — all identical in
  shape. No web-app call site changed.
- Behavioral changes verified live against the built core:
  - Blueprint stamps the marker, not the sentinel:
    `title: !must_fill Hello # string`, `count: !must_fill # string`,
    `bio: !must_fill # markdown` — one-space ` # ` gap, no `<must-fill>`, no
    `; delete-ok`.
  - An incomplete document validates clean: omitted Unendorsed fields produce
    **no** diagnostics (`validation::field_absent` is gone).
  - A bare scalar in a string field (`count: 47`) coerces silently — no
    `type_mismatch`.
- web-app only *displays* diagnostics; it never gated render or doneness on
  `validation::field_absent` / `validation::must_fill_sentinel`, so the retired
  codes caused zero breakage. (A consumer that *did* gate on them would see the
  codes silently stop appearing — the guide calls this out correctly.)

## Friction, roughly in order of impact

### 1. The experimental build self-identifies as `0.92.1` — the version lies

`Cargo.toml` carries the *last released* version (`0.92.1`); CI bumps it only at
release. So `./scripts/build-wasm.sh` on `main` emits a package whose
`package.json` says `"version": "0.92.1"` — **byte-identical version to the
already-published release** the consumer is migrating *from*.

Consequences for a third-party consumer:
- Cannot `npm install @quillmark/wasm@0.93` — the 0.93 artifact does not exist
  under that version.
- `node_modules/@quillmark/wasm/package.json` reads `0.92.1`; nothing in the
  installed package distinguishes "0.93 experimental" from "0.92.1 released."
- A vendored `file:` install records `"version": "0.92.1"` in `package-lock.json`
  (see web-app's lock). The filename can say `0.93.0-experimental`; the manifest
  inside still says `0.92.1`.

This is the single biggest integration hazard: **version is the consumer's
contract, and the experimental build's contract is indistinguishable from the
prior release.** A pre-release version stamp on `main`
(e.g. `0.93.0-dev`/`0.93.0-rc.N`) would remove all of this ambiguity.

### 2. wasm-bindgen CLI must match the locked `0.2.118` exactly — and can't be bumped

A fresh environment ships a newer `wasm-bindgen-cli` (`0.2.122`). The build
fails immediately:

```
rust Wasm file schema version: 0.2.118
   this binary schema version: 0.2.122
```

The obvious fix — bump the lock to `0.2.122` — is blocked: `js-sys 0.3.95` (a
transitive dep) hard-pins `wasm-bindgen =0.2.118`. So the only path is to pin the
**CLI** down to match:

```
cargo install -f wasm-bindgen-cli --version 0.2.118   # ~1.5 min from source
```

`scripts/build-wasm.sh` mentions `0.2.118` in a comment, but neither the
migration guide nor a top-level build doc states the exact-match requirement up
front, and the script does not detect/assert the CLI version before running.
Suggestion: have `build-wasm.sh` read the locked `wasm-bindgen` version and
fail early with the exact `cargo install` line if the CLI differs.

### 3. `wasm32-unknown-unknown` target not assumed

`rustup target add wasm32-unknown-unknown` was required before the first build.
Expected for a from-source build, but worth one line in build prerequisites.

### 4. The `render` → `typst` feature rename is a no-op for script users — but reads as an action item

The guide's "**Action (from-source WASM builders only):** replace
`--features render` with `--features typst`" implies work. But the canonical
`scripts/build-wasm.sh` never passed `--features render`; it uses default
features (typst), `--no-default-features --features pdfform-preview`, and
`--no-default-features` (core). Anyone building via the provided script needs to
do **nothing**. The action only applies to hand-rolled `cargo build` invocations.
Worth scoping the note to "if you invoke cargo directly" to avoid a phantom todo.

### 5. Artifact size, and the `--ci` trap

Release (`wasm-release`) sizes: typst backend 26 MB raw / **8.63 MB gzip**; full
packed tarball **12 MB**. The fast `--ci` profile is unoptimized: typst backend
22 MB **gzip**, packed tarball **31 MB** — ~2.5× larger. For a consumer who
vendors an experimental build, this matters: `--ci` is for correctness checks
only; vendor/ship the default release build. The size difference is not flagged
where a hurried integrator would see it.

### 6. `pkg/.gitignore` ignores everything — no in-repo experimental artifact

The build writes `pkg/` with a `.gitignore` of `*`. Correct for generated output,
but it means there is no committable experimental artifact in-tree; the only ways
to hand 0.93 to a consumer are `npm publish` (gated on a release) or a manual
`npm pack` + vendor. For this integration the artifact was `npm pack`ed and
vendored into `web-app/vendor/`.

### 7. Engine floor: node ≥ 22 (package) vs node ≥ 24 (web-app)

`@quillmark/wasm` sets `engines.node >=22`. web-app pins `>=24` with
`engine-strict=true`, and the environment had only node 20–22; node 24 had to be
installed before `npm install` would proceed. Not a quillmark defect, but the
package's own node floor is real and stricter than some SvelteKit baselines —
worth stating in the README for consumers on older runtimes.

### 8. Could not exercise the browser test project (environment, not quillmark)

web-app's `client` vitest project drives Playwright, which expects its pinned
`chromium-1228` build; the environment's pre-installed Chromium did not match, so
the in-browser wasm tests did not run here. The `server` project (which also
loads the real wasm) ran fully. Flagged only so the verification scope is honest.

## web-app changes made for this integration

- `package.json` / `package-lock.json`: `@quillmark/wasm` → the vendored
  experimental tarball (`file:vendor/quillmark-wasm-0.93.0-experimental.tgz`).
- `vendor/quillmark-wasm-0.93.0-experimental.tgz`: the packed release build of
  `@quillmark/wasm` from quillmark `main`.
- `src/lib/utils/schema-utils.ts`: doc comment updated — `<must-fill>` sentinel
  and `validation::field_absent` (both retired in 0.93) → the `!must_fill` marker
  and `validation::must_fill` warning model.
- `src/lib/services/quillmark/diagnostic-utils.test.ts`: the path-preservation
  sample used the retired `validation::field_absent` code; switched to the live
  `validation::must_fill` warning so the fixture reflects a code 0.93 can emit.

No call-site or logic changes were required.
