# Phase 0 — Spike B finding: source-map inversion + navigation

Status: **reported.** Gates the phase-2 emit + source-map design. Runnable
evidence: `crates/richtext-spikes/tests/spike_b_sourcemap.rs` (6 assertions).

## Question

Does glyph-offset inversion yield a usable point↔corpus map through
`escape_markup`, at cluster granularity, on real typeset output — and is the
per-run map invertible by *recomputation*, with no per-character stored tables?

## Answer

**Yes, cluster-exact, recomputation-only** — with one named coupling to handle.

### The escape transform is character-local except one 2-char coupling

`quillmark_typst::convert::escape_markup` is a per-char map (`*`→`\*`, `#`→`\#`,
…, a plain char → itself) **except** the `//`→`\/\/` rule, a 2-char lookahead.
`sourcemap.rs` reconstructs the per-char generated spans in one left-to-right
scan that absorbs the coupling (a `/` followed by `/` emits `\/`); its output is
cross-checked byte-for-byte against the shipped `escape_markup` over a battery
covering every special char, `//`, multibyte, and astral clusters
(`reimplemented_escape_matches_the_shipped_escaper`). **That cross-check is the
run-alignment tripwire the plan asks for**: if the backend grows an escape rule
the scan doesn't mirror, the test fails. No other escape construct breaks
alignment.

### The map is invertible by recomputation, no stored tables

Each corpus char maps to a contiguous generated byte span. `forward(char)` →
byte range and `invert(byte)` → char both come from cumulative span lengths that
are a **pure function of the run text** — rebuilding from the same text yields
an identical map, and nothing is carried between runs
(`map_needs_no_per_character_table_only_the_run_text`). A real emitter recomputes
the transform per text run; it need not persist a character table.

### Granularity is cluster-exact, never sub-char

A corpus char that escapes to several bytes (`*`→`\*`; `你`→3 UTF-8 bytes;
`😀`→4) is one cluster: **every** byte of its generated span inverts to that
single char (`inversion_is_cluster_exact_not_sub_char`). There is no sub-char
position a caret could occupy, so a page hit lands on the char, never between
the backslash and the star. This confirms the plan's expectation — the design
should say **cluster-exact**, not "character-exact", everywhere it means this.

### The span→byte→field chain resolves on real typeset output

`real_render_resolves_a_point_to_the_content_field` drives the actual Typst
backend on a body full of escape-worthy characters, then uses the public
`regions()`/`field_at()`: the field surfaces a positive-area box and a point
inside it resolves back to `body`. This exercises the shipped chain —
`glyph.span.0` → `DiagSpan`/`Source::range` → byte range → field-window
containment (`span_scan.rs`, `Classifier::classify`).

### Origin-less ink is nav-ignored

A point in the margin, off all field ink, resolves to nothing
(`origin_less_ink_and_empty_space_are_nav_ignored`) — the forward-direction
counterpart to "list markers / package numbering / plate decorations are
nav-ignored". Ink whose span resolves outside every content window is not
misattributed.

## The remaining phase-2 build step (scoped, not a red flag)

Today's `Classifier` resolves `glyph.span.0` to a *node* byte range and tests
window **containment** — field granularity, which is all the shipped `regions()`
needs. Character-precision navigation adds exactly one mechanical step, and its
raw material is already in hand:

1. resolve `glyph.span.0` to the node's generated byte range (existing);
2. add `glyph.span.1` — the intra-node byte offset **unused today at
   `span_scan.rs:197`** — to the range start, giving the exact generated byte;
3. invert that byte through this run map (proven cluster-exact) to a corpus
   char.

Steps 1 and 3 are validated here (chain resolution on real output; escape
inversion). Step 2 is reading a `u16` already present on every glyph the walk
touches — no new Typst capability, no per-render parse (the model's whole point:
the source map is a codegen artifact, built once at emit, not re-derived from a
re-parse). This is phase-2 emit work, scoped separately, and it is unblocked.

## Explicitly out of scope

The `regions()` run-machine rework for highlight boxes (grounding §3.2) — that
is phase-2 work. This spike touches navigation only.
